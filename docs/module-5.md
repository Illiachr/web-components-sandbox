**Модуль 5: Жизненный цикл, Оптимизация и Деплой**

В этом модуле мы заставим компоненты гибко реагировать на внешние изменения, оптимизируем рендеринг, чтобы приложение не «моргало» при обновлениях, и подготовим его к публикации.

---

## 1. Теоретическая база

До этого момента мы использовали только один хук жизненного цикла — `connectedCallback()`. Но у нативной платформы есть еще пара мощных инструментов:

1. **`attributeChangedCallback(name, oldValue, newValue)`** — этот хук вызывается каждый раз, когда меняется HTML-атрибут компонента. Он позволяет компоненту реактивно обновляться, если его изменили снаружи через JS или прямо в HTML (например, `<profile-list theme="dark">`).
2. **`static get observedAttributes()`** — массив строк, в котором мы обязаны явно перечислить, за какими именно атрибутами браузер должен следить. Если атрибута нет в этом списке, хук выше на него не отреагирует.
3. **`disconnectedCallback()`** — вызывается, когда компонент удаляется из DOM. Здесь мы обязаны **удалять глобальные слушатели** (`window.removeEventListener`), иначе приложение начнет «течь» по памяти.

## 2. Практическая реализация и паттерны Модуля 5

### Шаг 1: Добавляем реактивность через атрибуты

Давай сделаем так, чтобы наше приложение могло динамически менять, например, заголовок или фильтр. Вот как выглядит паттерн синхронизации атрибутов со стейтом:

```javascript
class ProfileForm extends HTMLElement {
  // 1. Указываем, за какими атрибутами следить
  static get observedAttributes() {
    return ['mode']; // например: mode="create" или mode="edit"
  }

  // 2. Реагируем на изменения
  attributeChangedCallback(name, oldValue, newValue) {
    if (oldValue === newValue) return;
    
    if (name === 'mode') {
      this._state.mode = newValue;
      this.render(); // Перерисовываем компонент при смене режима!
    }
  }
}

```

---

### Шаг 2: Защита от утечек памяти (`disconnectedCallback`)

Поскольку в Модуле 4 мы начали активно вешать слушатели на глобальный объект `window` (события `app-navigate` или `popstate`), при удалении компонентов эти слушатели останутся в памяти браузера навсегда.

Каждый раз, когда роутер будет пересоздавать или скрывать компоненты, старые функции будут продолжать выполняться. Нам нужно это исправить:

```javascript
export class AppRouter extends HTMLElement {
  constructor() {
    super();
    // Привязываем контекст, чтобы метод удаления сработал корректно
    this.boundPopstate = () => this.handleRoute();
  }

  connectedCallback() {
    window.addEventListener('popstate', this.boundPopstate);
  }

  // Чистим за собой мусор!
  disconnectedCallback() {
    window.removeEventListener('popstate', this.boundPopstate);
  }
}

```

### Шаг 3: Оптимизация рендеринга (Паттерн Debounce / Шаблонизация)

Сейчас при любом изменении стейта мы вызываем метод `render()`, который заново переписывает DOM-узлы. Если данных много, это может вызвать просадку FPS.

Для оптимизации мы внедрим паттерн **Debounce для рендеринга** с помощью `requestAnimationFrame`. Это гарантирует, что даже если стейт обновится 1000 раз за миллисекунду, физический рендеринг в DOM произойдет ровно 1 раз в следующий кадр анимации монитора.

```javascript
// Внутри базового компонента или формы:
updateState(newState) {
  this._state = { ...this._state, ...newState };
  
  // Если рендеринг уже запланирован — отменяем старый
  if (this._renderPending) {
    cancelAnimationFrame(this._renderPending);
  }

  // Планируем рендеринг на следующий кадр (60-120 FPS)
  this._renderPending = requestAnimationFrame(() => {
    this.render();
    this._renderPending = null;
  });
}

```

## 3. Деплой приложения

Поскольку наше приложение написано на **чистых ES-модулях**, нам не нужны Webpack, Vite или TurboPack для того, чтобы запустить его в продакшене. Браузеры уже умеют всё сами!

### Как подготовить проект к деплою:

1. Убедиться, что все пути к файлам в импортах указаны *относительно* и содержат расширения файлов (например, `import './app-link.js';`, а не `import 'app-link'`).
2. Проверить конфигурацию роутера: для работы `pushState` на реальном хостинге (например, GitHub Pages, Vercel или Netlify) сервер должен перенаправлять все запросы (Fallback) на `index.html`.
* *Для Vercel/Netlify достаточно добавить простой файл конфигурации `vercel.json` или `_redirects` с правилом: `/* /index.html 200`.*
